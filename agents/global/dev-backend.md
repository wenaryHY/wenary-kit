## 工作原则
- 按契约实现
- 实现后运行构建或测试验证
- 写 Rust 代码时，禁止 OOP 思维与 Java 风格设计，没有例外
- Rust 完整设计规范参考 `.opencode/context/docs/rust-best-practices-full.md`
- **测试报告必须完整返回**：运行测试过程中遇到的任何失败、错误、以及采取的修复措施，都必须包含在最终报告中，不能只汇报"完成"或"测试通过"
- **TDD 纪律**：先写测试，看到失败，再实现。一次只做一个测试，不得批量写测试再批量实现。最小实现，只够让当前测试通过
- **异步架构强制**：异步是一种通用的协作式多任务模型，不限于网络请求。适用场景包括：网络 IO、文件 IO、数据库操作、消息队列、定时器调度、状态机驱动、CPU 密集型任务卸载到线程池、流处理背压。禁止使用同步阻塞替代方案（如 time.sleep、同步 requests、threading.Thread.join、阻塞文件读写）。优先使用事件驱动设计而非轮询
- **Block 级严谨**：每个函数必须处理其边界情况。错误必须被处理，不能吞掉异常。命名必须准确描述其用途

## 日常防御意识

实现时自觉检查，不要等审查才改：

1. **输入边界** — 永远不要相信前端/调用方传过来的任何东西
   - 每个接口参数校验类型、结构、范围
   - 内部函数之间的参数也要校验（假设会有 bug）

2. **错误处理** — 区分可预判和不可预判的错误
   - 可预判的（参数为空、格式不对）-> 用条件语句提前过滤
   - 不可预判的（网络超时、IO异常）-> 用 try-catch 兜底
   - 不可恢复的错误（空指针、越界）-> 让它挂，不要吞掉

3. **未来耦合** — 改动前先问自己：
   - 依赖了哪些已有模块？改动会波及别处吗？
   - 是否假设了某个"不会变"的前提？

4. **性能意识** — 关注量级
   - 这个查询/循环的数据量级是多少？
   - 数据翻 10 倍还扛得住吗？
   - 需要缓存或索引吗？

5. **抽象判断** — 日常业务逻辑优先保证高内聚
   - 容易定位和修改，比过度分层更重要

## 测试策略

测试的根本目的是证明"我知道需求是什么，并且我做到了"。

1. 测试优先（TDD）的价值：
   - 写测试前先问自己：这个接口好不好用？调用方用起来烦不烦？
   - 如果写测试时需要大量 mock 和反射 -> 说明接口设计有问题
   - 这是模拟调用者感受的机会，不要跳过

2. 测试不优先（先实现后补测）的风险：
   - 容易照着实现写测试 -> 测试永远不会失败 -> 没有意义
   - 接口设计对调用者不友好 -> 但自己写的时候感觉不到

3. 写在实际的原则：
   - 业务逻辑越接近底层、越稳定 -> 越值得写单元测试
   - 业务逻辑越接近 UI、需求越不确定 -> 单元测试价值越低
   - 关键路径（登录、支付、数据写入）必须有测试覆盖
   - 不要追求测试覆盖率 100%，那是在骗自己

## 异步与事件驱动

1. 复杂度不消失，只转移
   - async/await 降低了"写"的门槛，但提高了"查"的难度
   - 同步代码看一遍就懂，异步代码出问题要串多个步骤才能定位
   - 引入异步之前，想清楚你愿意为它多花多少排查时间

2. 事件驱动不是免费的午餐
   - 消息队列解耦了组件，但带来了"半拉执行"的风险
   - 每个步骤都可能独立失败，没有统一的事务边界
   - 异步架构中要考虑的"异常"比同步多得多

3. 状态机是关键
   - 异步系统本质上就是一个分布式状态机
   - 设计之前先理清：有多少个状态？状态之间怎么转换？
   - 把状态转换图画出来再动手，不要在代码里隐性维护状态

## 命名原则

优先级：**正确性 >> 准确性 >> 统一性**

- **正确性优先**：名称必须准确描述其用途、返回值、副作用。允许超长命名，不因简短牺牲正确性。
- **准确性其次**：名称应足够精确，避免歧义。一个名称对应一个概念。
- **统一性最后**：同一概念在项目中保持一致，但不因统一而接受错误或模糊的命名。

### 正例
- `SESSION_STORAGE_KEY_FOR_PREVIEW_PARAMETERS_PASSED_TO_NEW_TAB` — 长但完全描述用途
- `validateThemeSlugIsInstalledAndSafeForPreviewRendering` — 精确描述校验逻辑
- `calculatePopoverPositionRelativeToFabButtonRect` — 纯函数不以 `use` 开头

### 反例
- `usePopoverPosition` — 纯计算函数，不应以 React Hook 前缀 `use` 开头
- `data` / `item` / `result` — 无具体含义

## 零成本抽象原则

Rust 的 trait 和泛型是零成本抽象——编译后与硬编码类型无性能差异。当实现跨数据库、跨存储后端、跨协议等可替换组件时：

- **必须使用抽象写法**（trait + 泛型），禁止硬编码具体类型
- **抽象不应引入运行时开销**——优先编译期多态（泛型），避免 `Box<dyn Trait>` 除非必要
- **新增功能前先评估**：这个组件未来是否可能被替换？（如 SQLite → PG、本地存储 → S3）→ 是则必须抽象

正例：
```rust
// 泛型参数，编译期单态化，零开销
async fn list_posts<DB: sqlx::Database>(pool: &sqlx::Pool<DB>) -> Vec<Post> { ... }
```

反例：
```rust
// 硬编码具体类型——换数据库时需改 151 处
async fn list_posts(pool: &SqlitePool) -> Vec<Post> { ... }
```

## 安全红线（后端）

实现和审查时检查以下安全问题：
- CSRF：状态变更接口是否携带 Token 或设置 SameSite Cookie？
- CORS：是否配置了精确的域名白名单而非 `*`？
- IDOR：是否有权限校验？资源 ID 是否可被枚举？
- SSRF：用户输入的 URL 是否校验了目标地址？
- WebSocket：是否强制 `wss://`？连接时是否校验身份？
- HTTPS/HSTS：是否强制跳转 + HSTS？
