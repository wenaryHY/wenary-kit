## 前端设计缺陷导致的安全漏洞 — 经验手册

> 本文档系统梳理了因前端架构设计、编码习惯或配置不当而引发的安全漏洞，按攻击面分类，每条包含原理、典型场景、危害等级与防御要点。供开发团队在设计和评审阶段对照排查。

---

### 一、注入类漏洞

#### 1.1 反射型 XSS（Cross-Site Scripting）

**原理：** 前端将 URL 参数、表单数据等用户输入直接拼接到页面 HTML 中渲染，未做转义或过滤。攻击者构造含恶意脚本的链接，诱导受害者点击后脚本在浏览器中执行。

**典型场景：**
- 搜索框回显关键词时直接 `innerHTML = searchQuery`
- 错误提示页拼接 `?msg=` 参数内容

**危害等级：** 高 — 可窃取 Cookie/Session、劫持用户操作、钓鱼

**防御要点：**
- 所有动态内容输出前做 HTML 实体编码（`<` → `&lt;` 等）
- 使用框架自带的模板转义机制（Vue 的 `{{ }}`、React 的 JSX 默认转义）
- 富文本场景使用 DOMPurify 等白名单过滤器
- 配置 CSP 限制内联脚本和外部脚本来源

#### 1.2 存储型 XSS

**原理：** 用户提交的恶意内容被服务端持久化存储（数据库、文件），其他用户浏览该页面时自动加载执行。

**典型场景：**
- 评论区、论坛帖子、用户个人主页
- 富文本编辑器未过滤 `<script>` 或事件属性（`onerror`、`onload`）

**危害等级：** 严重 — 影响面广，所有访问该页面的用户均受害

**防御要点：**
- 输入端：白名单过滤允许的 HTML 标签和属性
- 存储端：对特殊字符做编码后存储
- 输出端：根据上下文（HTML/JS/URL/CSS）选择对应编码方式
- 关键 Cookie 设置 `HttpOnly`，阻止脚本读取

#### 1.3 DOM-based XSS

**原理：** 恶意数据从"源"（source，如 `location.hash`、`document.referrer`、`window.name`）流向"汇"（sink，如 `eval()`、`innerHTML`、`document.write()`），全程在客户端完成，不经过服务端。

**典型场景：**
- 根据 `location.hash` 切换页面内容时直接拼接 HTML
- 使用 `document.write(location.search)` 渲染页面
- `eval()` 解析 URL 参数中的 JSON 数据

**危害等级：** 高 — 隐蔽性强，传统服务端 WAF 难以检测

**防御要点：**
- 避免使用危险 sink：`eval()`、`innerHTML`、`document.write()`、`setTimeout(string)`
- 必须操作 DOM 时使用 `textContent` 替代 `innerHTML`
- 对 URL 参数做严格校验和编码后再使用
- CSP 中禁止 `unsafe-eval`

#### 1.4 原型链污染（Prototype Pollution）

**原理：** 攻击者通过可控的属性赋值（如 `obj[key] = value`，其中 key 来自用户输入），向 `Object.prototype` 注入属性，影响所有 JavaScript 对象。

**典型场景：**
- 前端深度合并（deep merge）函数未校验 `__proto__`、`constructor`、`prototype` 等危险键
- 从 URL 参数解析配置对象时未做键名过滤

**危害等级：** 高 — 可导致逻辑篡改、结合其他漏洞升级为 XSS 或 RCE（Node.js 环境）

**防御要点：**
- 使用 `Object.create(null)` 创建无原型的纯净对象作为容器
- 合并/赋值前校验键名，拒绝 `__proto__`、`constructor`、`prototype`
- 使用 `Map` 替代普通对象做键值映射
- 对第三方库的 merge 函数保持警惕，及时升级已知有漏洞的版本

### 二、跨站请求伪造（CSRF）

**原理：** 浏览器在发送请求时自动携带目标站点的 Cookie，攻击者利用这一机制，在恶意页面中构造请求（表单提交、`<img>` 标签、`fetch`），冒充已登录用户执行非预期操作。

**典型场景：**
- 转账页面只依赖 Cookie 鉴权，无额外校验
- 用户登录状态下访问恶意页面，页面自动提交转账表单

**危害等级：** 高 — 可在用户不知情时执行敏感操作（转账、改密、删数据）

**防御要点：**
- 所有状态变更请求携带 CSRF Token，服务端校验
- Cookie 设置 `SameSite=Strict` 或 `SameSite=Lax`
- 关键操作增加二次验证（密码、短信验证码、生物识别）
- 校验请求头 `Referer` / `Origin` 是否来自合法域名

---

### 三、点击劫持（Clickjacking / UI Redressing）

**原理：** 攻击者用透明 `<iframe>` 将目标网站嵌入恶意页面，覆盖在诱饵内容之上。用户以为点击的是可见按钮，实际点击了 iframe 中的隐藏元素。

**典型场景：**
- 透明 iframe 覆盖"领取奖品"按钮，实际触发的是"确认转账"
- 多层 iframe 嵌套（frame splitting）绕过简单检测

**危害等级：** 中高 — 可劫持任意用户操作

**防御要点：**
- 响应头设置 `X-Frame-Options: DENY` 或 `SAMEORIGIN`
- CSP 中配置 `frame-ancestors 'self'` 限制可嵌入的域名
- 前端 JS 检测 `window.top !== window.self` 并阻断渲染（作为辅助手段，不可单独依赖）

### 四、敏感信息泄露

#### 4.1 前端代码中硬编码密钥

**原理：** API Key、Secret、数据库凭据等直接写在前端 JS 代码或配置文件中，任何人查看源码即可获取。

**典型场景：**
- 前端代码中直接写 `const API_KEY = "sk-xxxxx"`
- `.env` 文件中的 `REACT_APP_SECRET` 被打包进生产构建
- Source Map 文件（`.map`）暴露到生产环境，泄露完整源码

**危害等级：** 严重 — 密钥一旦暴露即视为已泄露

**防御要点：**
- 所有密钥通过后端代理访问，前端永远不接触敏感凭据
- 生产构建移除 Source Map（`devtool: false` 或上传到内部平台）
- 使用环境变量时区分 `PUBLIC_` 前缀（可暴露）和无前缀（仅服务端可用）
- 定期扫描代码仓库中的密钥泄露（工具：git-secrets、truffleHog）

#### 4.2 错误信息泄露

**原理：** 生产环境向前端返回详细的错误堆栈、SQL 语句、内部路径等信息，暴露系统架构。

**典型场景：**
- API 返回 `500` 时附带 Java 堆栈跟踪
- 数据库错误信息直接返回到前端页面

**危害等级：** 中 — 为攻击者提供系统内部信息，降低攻击门槛

**防御要点：**
- 生产环境统一返回通用错误提示（如"服务器繁忙"）
- 详细日志仅写入服务端日志系统
- 前端捕获异常后不向用户展示技术细节

#### 4.3 浏览器缓存与自动补全泄露

**原理：** 浏览器自动缓存页面、表单自动补全功能可能将敏感数据（密码、身份证号、银行卡号）保存在本地，其他使用同一设备的用户可获取。

**典型场景：**
- 密码输入框未禁用自动补全，公共电脑上被下一位用户看到
- 敏感页面被浏览器缓存，离线即可从缓存中读取

**危害等级：** 中 — 物理接触设备即可利用

**防御要点：**
- 敏感输入框设置 `autocomplete="off"`（注意：部分浏览器可能忽略）
- 敏感页面响应头设置 `Cache-Control: no-store`
- 密码字段使用 `autocomplete="new-password"`（创建场景）或 `"current-password"`（登录场景）配合密码管理器
- 表单提交后清除敏感字段值

### 五、不安全的存储设计

#### 5.1 JWT/Token 存储在 localStorage

**原理：** 将认证 Token 存放在 `localStorage` 中，任何同源 JavaScript 均可读取。一旦存在 XSS 漏洞，攻击者可直接窃取 Token。

**危害等级：** 高 — Token 被盗意味着完全冒充用户身份

**防御要点：**
- 优先使用 `HttpOnly` + `Secure` + `SameSite` Cookie 存储 Token
- 若必须使用 localStorage，配合短期 Access Token + 长期 Refresh Token（Refresh Token 存 HttpOnly Cookie）
- 实施 Token 绑定（如设备指纹），被盗后在其他设备无法使用

#### 5.2 敏感数据明文存储在客户端

**原理：** 将用户隐私数据（身份证号、手机号、银行卡号）以明文形式存储在 `localStorage`、`sessionStorage` 或 Cookie 中。

**危害等级：** 高 — 任何 XSS 或物理接触均可读取

**防御要点：**
- 前端不存储不必要的敏感数据
- 必须存储时使用 `sessionStorage`（关闭标签页即清除）并配合加密
- Cookie 设置 `HttpOnly`、`Secure`、`SameSite`

---

### 六、开放重定向（Open Redirect）

**原理：** 前端根据 URL 参数决定跳转目标，未校验目标地址的合法性。攻击者构造形如 `https://trusted.com/redirect?url=https://evil.com` 的链接，利用用户对可信域名的信任实施钓鱼。

**典型场景：**
- 登录成功后跳转到 `?redirect=` 参数指定的地址
- "返回首页"按钮链接到参数化的外部 URL
- 404 页面自动跳转到用户传入的地址

**危害等级：** 中高 — 常被用作钓鱼攻击链的一环

**防御要点：**
- 维护合法跳转域名白名单，不在白名单中的地址拒绝跳转
- 仅允许相对路径跳转（以 `/` 开头，不以 `//` 开头）
- 对参数值做 URL 解码后校验，防止编码绕过
- 避免直接使用用户输入作为跳转目标

---

### 七、CORS 配置不当

**原理：** 服务端将 `Access-Control-Allow-Origin` 设置为 `*` 或动态读取请求的 `Origin` 头，导致任意第三方网站可以跨域读取本应用的敏感数据。

**典型场景：**
- API 响应头 `Access-Control-Allow-Origin: *` 且 `Access-Control-Allow-Credentials: true`（浏览器会拒绝此组合，但仅设 `*` 仍可泄露非凭证数据）
- 动态反射 Origin：`Access-Control-Allow-Origin: <request Origin>`，等同于完全开放

**危害等级：** 高 — 攻击者可在自己的网站上读取用户的私密 API 响应

**防御要点：**
- 严格配置允许的域名白名单，不使用通配符
- 需要携带 Cookie 的跨域请求，白名单中精确列出可信域名
- 区分公开 API（无需认证，可适度开放）和私有 API（严格限制）

---

### 八、接口越权与 IDOR

#### 8.1 前端权限控制形同虚设

**原理：** 仅在前端通过 UI 隐藏（如不渲染管理员按钮）来控制权限，后端接口未做独立的权限校验。攻击者直接调用 API 即可越权操作。

**典型场景：**
- 普通用户看不到"删除"按钮，但直接 POST `/api/users/delete` 成功
- 前端根据角色隐藏菜单，但路由和 API 未做对应校验

**危害等级：** 严重 — 权限控制的核心不在前端

**防御要点：**
- 后端对每个接口独立校验当前用户的角色和权限
- 前端权限控制仅作为 UX 优化，不可作为安全屏障
- 使用 RBAC/ABAC 模型在后端统一管控

#### 8.2 不安全的直接对象引用（IDOR）

**原理：** URL 或 API 参数中直接暴露资源的数字 ID（如 `/api/orders/123`），攻击者通过递增或遍历 ID 访问他人数据。

**典型场景：**
- `/api/users/42/profile` → 改为 `/api/users/43/profile` 查看他人信息
- 订单号使用自增 ID，遍历即可获取所有订单

**危害等级：** 高 — 可批量泄露用户数据

**防御要点：**
- 后端校验请求者是否有权访问目标资源
- 使用不可预测的标识符（UUID、Hashids）替代自增 ID
- 实施速率限制，检测异常的批量请求模式

### 九、认证与验证码设计缺陷

#### 9.1 验证码仅前端校验

**原理：** 验证码的正确答案以明文或简单加密形式存在于前端代码中，或校验逻辑仅在客户端执行。

**危害等级：** 高 — 验证码完全失效

**防御要点：**
- 验证码校验必须在服务端完成
- 验证码与用户会话绑定，防止替换目标账户
- 验证码使用后立即失效（一次性）
- 限制尝试次数，失败过多后锁定或更换

#### 9.2 验证码设计弱点

**原理：** 验证码复杂度不足、无过期机制、无频率限制，使其可被暴力破解或 OCR 识别。

**典型场景：**
- 4 位纯数字验证码，仅 10000 种组合
- 验证码长期有效不过期
- 静态图片验证码，可被 AI 批量识别

**防御要点：**
- 使用 6 位以上字母+数字混合，加入干扰线和扭曲
- 设置短有效期（如 60 秒）和使用后即失效
- 连续失败 5 次后强制更换或增加图形验证
- 考虑使用行为验证（滑动拼图、点选）替代传统字符验证码

#### 9.3 密码重置流程缺陷

**原理：** 密码重置链接或验证码在前端可预测、可重放、或未与目标账户绑定。

**典型场景：**
- 重置 Token 使用用户 ID 的 MD5（可预测）
- 重置链接中的 Token 可被多次使用
- 验证码发送后，修改请求中的手机号即可重置他人密码

**防御要点：**
- 重置 Token 使用密码学安全的随机数（`crypto.randomUUID()`）
- Token 一次性使用，使用后失效
- Token 与目标账户绑定，服务端校验一致性
- 设置短有效期（如 15 分钟）

### 十、postMessage 跨域通信漏洞

**原理：** 使用 `window.postMessage()` 进行跨窗口/iframe 通信时，未校验消息来源（`origin`）或未指定目标域（使用 `*`），导致恶意页面可注入或窃取数据。

**典型场景：**
- 监听 `message` 事件时不校验 `event.origin`
- 发送消息时使用 `postMessage(data, '*')`，任何嵌入的 iframe 都能接收
- 接收到消息后直接 `eval()` 或 `innerHTML` 渲染

**危害等级：** 高 — 可跨域窃取数据或注入恶意内容

**防御要点：**
- 接收端严格校验 `event.origin` 是否在白名单中
- 发送端明确指定目标域名，不使用 `*`
- 对消息内容做类型校验和转义，不直接执行
- 使用结构化克隆算法（不传字符串，传对象）减少注入面

### 十一、正则表达式拒绝服务（ReDoS）

**原理：** 前端使用的正则表达式存在嵌套量词或歧义结构（如 `(a+)+`、`(a|a)*`），攻击者构造特殊输入使正则引擎陷入指数级回溯，导致浏览器主线程阻塞。

**典型场景：**
- 前端表单校验使用复杂正则（如邮箱、URL 验证）
- 路由匹配库（如 `path-to-regexp`）的已知漏洞

**危害等级：** 中高 — 可导致页面无响应，影响用户体验；Node.js 服务端同受影响时可导致服务不可用

**防御要点：**
- 避免嵌套量词和交替中的重叠模式
- 使用在线工具（如 regex101、safe-regex）检测正则安全性
- 设置正则匹配的超时限制
- 优先使用库提供的验证函数而非手写复杂正则

---

### 十二、第三方依赖与供应链攻击

#### 12.1 使用含已知漏洞的依赖

**原理：** 引入的 npm 包、CDN 资源或第三方 SDK 存在已公开的安全漏洞（CVE），未及时更新。

**典型场景：**
- 旧版 jQuery 存在 XSS 漏洞（`$('<div>').html(userInput)`）
- lodash 的 `_.merge()` 存在原型链污染
- event-stream 包被植入恶意代码（2018 年真实事件）

**危害等级：** 严重 — 影响面取决于依赖的广泛程度

**防御要点：**
- 定期运行 `npm audit` / `pnpm audit` 扫描漏洞
- 使用 Snyk、Dependabot 等工具自动监控和修复
- 锁定依赖版本（`package-lock.json` / `pnpm-lock.yaml`）
- 对外部 CDN 资源使用 SRI（Subresource Integrity）校验：`<script integrity="sha384-..." crossorigin="anonymous">`

#### 12.2 CDN 资源劫持

**原理：** 从不安全的 CDN 加载 JS/CSS 资源，中间人可篡改内容注入恶意代码。

**危害等级：** 严重 — 等同于全站被植入后门

**防御要点：**
- 所有外部资源使用 HTTPS 加载
- 必须使用 SRI 校验资源完整性
- 关键资源自建 CDN 或镜像
- CSP 中限制可加载脚本的来源域名

### 十三、HTTPS 降级与中间人攻击

**原理：** 站点同时支持 HTTP 和 HTTPS，用户首次访问使用 HTTP，中间人在此阶段拦截并阻止 HTTPS 跳转，或注入恶意内容。

**典型场景：**
- 用户手动输入域名（不带 `https://`），默认走 HTTP
- 首次访问的 HTTP 请求被运营商注入广告脚本

**危害等级：** 高 — 所有后续通信均不安全

**防御要点：**
- 全站启用 HTTPS，HTTP 301 重定向到 HTTPS
- 响应头设置 `Strict-Transport-Security: max-age=31536000; includeSubDomains; preload`
- 将域名提交到 HSTS Preload List（浏览器内置 HTTPS 列表）
- 使用 Certificate Transparency 监控证书签发

---

### 十四、文件上传/下载漏洞

#### 14.1 前端文件类型校验不足

**原理：** 仅在前端通过文件扩展名或 MIME 类型校验上传文件，攻击者绕过前端直接调用 API 上传恶意文件。

**典型场景：**
- 前端只允许 `.jpg`、`.png`，但后端未做校验
- 上传的文件名未做清理，包含路径遍历字符（`../../etc/passwd`）

**危害等级：** 严重 — 可上传 WebShell 或恶意脚本

**防御要点：**
- 后端校验文件魔数（Magic Bytes），不依赖扩展名
- 上传目录禁止执行权限
- 文件名做随机化处理，不保留原始文件名
- 限制上传大小和频率

#### 14.2 前端目录遍历

**原理：** 下载接口参数可控，攻击者通过 `../` 遍历服务器目录，下载敏感文件。

**危害等级：** 严重 — 可读取服务器任意文件

**防御要点：**
- 后端对文件路径做规范化处理，拒绝包含 `..` 的路径
- 限制下载目录在指定范围内
- 使用文件名白名单而非参数直接拼接路径

---

### 十五、前端路由与权限绕过

**原理：** SPA 应用仅在前端路由层做权限守卫（如 Vue Router 的 `beforeEach`），但后端未对对应的 API 做权限校验。攻击者直接访问 API 或修改前端路由守卫逻辑即可越权。

**典型场景：**
- 前端路由守卫阻止跳转到 `/admin`，但 `/api/admin/*` 接口未做鉴权
- 用户通过浏览器 DevTools 修改路由守卫逻辑后直接访问受限页面

**危害等级：** 高 — 前端路由守卫可被轻易绕过

**防御要点：**
- 前端路由守卫仅用于 UX（隐藏菜单、友好提示），不可作为安全控制
- 后端对每个路由和 API 独立做权限校验
- 前端页面加载时向后端请求权限数据，无权限则返回 403

---

### 十六、WebSocket 安全缺陷

**原理：** 使用 `ws://` 而非 `wss://` 协议，通信内容明文传输；或 WebSocket 连接未做身份校验，任意客户端均可连接。

**危害等级：** 高 — 实时通信数据被窃听或篡改

**防御要点：**
- 强制使用 `wss://`（WebSocket Secure）
- WebSocket 连接建立时校验身份（Token 或 Cookie）
- 服务端校验消息来源和内容合法性
- 实施连接速率限制

---

### 十七、JSONP 劫持

**原理：** 利用 `<script>` 标签不受同源策略限制的特性，通过 JSONP 接口跨域获取数据。若 JSONP 接口返回敏感数据且未做来源校验，恶意网站可直接读取。

**危害等级：** 中高 — 可跨域窃取用户数据

**防御要点：**
- 废弃 JSONP，使用 CORS 替代
- 若必须保留 JSONP，添加随机 Token 校验请求来源
- JSONP 接口不返回敏感数据

---

### 十八、SSRF（服务端请求伪造）的前端触发

**原理：** 前端参数（如图片 URL、回调地址）被后端直接用于发起请求，攻击者传入内网地址（`http://127.0.0.1`、`http://169.254.169.254` 元数据服务）探测或攻击内网。

**典型场景：**
- "图片预览"功能传入 URL，后端直接请求该 URL 渲染
- Webhook 回调地址由用户输入

**危害等级：** 严重 — 可探测内网服务、读取云元数据（AWS/GCP 密钥）

**防御要点：**
- 后端校验请求目标，禁止内网 IP 和保留地址段
- 使用 URL 白名单限制允许请求的域名
- 对响应内容做大小和类型限制
- 云环境中屏蔽元数据服务地址（`169.254.169.254`）

---

### 十九、HTTP 劫持与内容注入

**原理：** 运营商或中间节点在 HTTP 响应中注入广告脚本、弹窗代码或恶意重定向。由于 HTTP 不加密，中间节点可任意篡改响应内容。

**危害等级：** 高 — 用户看到的页面内容已不可信

**防御要点：**
- 全站 HTTPS + HSTS
- CSP 限制可加载脚本和资源的来源
- 使用 SRI 校验关键外部资源

---

### 二十、综合防御清单

| 防御层 | 措施 | 覆盖漏洞类型 |
|--------|------|-------------|
| **输入校验** | 白名单过滤、类型检查、长度限制 | XSS、注入、原型污染、SSRF |
| **输出编码** | 上下文编码（HTML/JS/URL/CSS） | XSS（反射/存储/DOM） |
| **CSP** | 限制脚本来源、禁止 inline eval | 所有 XSS 变体、数据注入 |
| **Cookie 安全** | HttpOnly + Secure + SameSite | XSS 后果减轻、CSRF |
| **认证增强** | CSRF Token、二次验证、短有效期 Token | CSRF、会话劫持 |
| **传输安全** | 全站 HTTPS + HSTS + SRI | 中间人、CDN 劫持、HTTP 降级 |
| **权限控制** | 后端独立鉴权、RBAC/ABAC | 越权、IDOR、路由绕过 |
| **存储安全** | 不存敏感数据、HttpOnly Cookie 存 Token | Token 窃取、数据泄露 |
| **依赖管理** | npm audit、SRI、版本锁定 | 供应链攻击、已知 CVE |
| **错误处理** | 生产环境隐藏技术细节 | 信息泄露 |
| **速率限制** | 限流、验证码、账户锁定 | 暴力破解、枚举、ReDoS |
| **安全头** | X-Frame-Options、X-Content-Type-Options、Permissions-Policy | 点击劫持、MIME 嗅探 |

### 附录：安全响应头速查

```
# 防止点击劫持
X-Frame-Options: DENY

# 防止 MIME 类型嗅探
X-Content-Type-Options: nosniff

# 强制 HTTPS
Strict-Transport-Security: max-age=31536000; includeSubDomains; preload

# 内容安全策略
Content-Security-Policy: default-src 'self'; script-src 'self'; style-src 'self' 'unsafe-inline'; img-src 'self' data: https:; frame-ancestors 'none'

# 限制浏览器功能
Permissions-Policy: camera=(), microphone=(), geolocation=()

# 防止信息泄露
X-XSS-Protection: 0  (现代浏览器已由 CSP 替代，设为 0 避免旧实现的副作用)
Referrer-Policy: strict-origin-when-cross-origin
```
